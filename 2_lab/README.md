# Лабораторная работа 2 — Redis-кэширование (KeyDB)

Приложение расширяет CRUD-сервис заметок из Лабораторной работы 1, добавляя кэш-прокси на базе KeyDB (Redis-совместимый сервер) по стратегии **cache-aside**.

## Запуск

```bash
docker compose up --build -d
```

После запуска сервис доступен на `http://localhost:8080`.

### HTTP-эндпоинты

| Метод  | Путь           | Описание                          |
|--------|----------------|-----------------------------------|
| GET    | /health        | Проверка работоспособности        |
| GET    | /notes         | Список заметок (с кэшем)          |
| POST   | /notes         | Создать заметку (инвалидирует кэш)|
| DELETE | /notes/{id}    | Удалить заметку (инвалидирует кэш)|

### Пример запросов

```bash
# Получить все заметки (первый запрос — MISS, повторный — HIT)
curl http://localhost:8080/notes

# Создать заметку
curl -X POST http://localhost:8080/notes \
  -H "Content-Type: application/json" \
  -d '{"title":"Заметка 1","body":"Текст заметки"}'

# Удалить заметку
curl -X DELETE http://localhost:8080/notes/1
```

## Теоретические вопросы

### 1. Что такое кэширование и зачем оно используется?

**Кэширование** — это техника временного хранения результатов дорогостоящих операций (обращений к БД, внешним API, сложных вычислений) в быстром промежуточном хранилище (кэше) с целью ускорить повторные запросы.

**Зачем используется:**
- Снижает задержку ответа: данные из кэша (оперативная память) читаются на несколько порядков быстрее, чем из реляционной БД или с диска.
- Уменьшает нагрузку на базу данных и бэкенд-сервисы, повышая их пропускную способность.
- Позволяет системе горизонтально масштабироваться без пропорционального роста числа запросов к хранилищу.
- Повышает отказоустойчивость: при временной недоступности БД кэш может обслуживать запросы.

---

### 2. Какие стратегии кэширования существуют?

| Стратегия | Описание | Когда применять |
|-----------|----------|-----------------|
| **Cache-Aside (Lazy Loading)** | Приложение само управляет кэшем: читает из кэша, при промахе идёт в БД и кладёт данные в кэш. | Читается чаще, чем пишется; данные не всегда нужны в кэше. |
| **Read-Through** | Кэш сам обращается в БД при промахе, прозрачно для приложения. | Однородный доступ; приложение не хочет знать об источнике. |
| **Write-Through** | При каждой записи данные сохраняются одновременно в кэш и в БД. | Требуется сильная согласованность; чтений много, а промахи нежелательны. |
| **Write-Behind (Write-Back)** | Запись идёт только в кэш, а в БД — асинхронно позже. | Высокая интенсивность записи; допустима небольшая потеря данных. |
| **Refresh-Ahead** | Кэш обновляется заранее, до истечения TTL, если ключ часто запрашивается. | Данные обновляются предсказуемо; критична минимальная задержка. |

В данной лабораторной работе реализована стратегия **Cache-Aside**.

---

### 3. Какие стратегии вытеснения кэша существуют?

Стратегия вытеснения определяет, какой элемент удаляется из кэша при нехватке памяти.

| Политика | Расшифровка | Принцип |
|----------|-------------|---------|
| **LRU** | Least Recently Used | Вытесняется элемент, к которому дольше всего не обращались. |
| **LFU** | Least Frequently Used | Вытесняется элемент, к которому обращались реже всего. |
| **FIFO** | First In, First Out | Вытесняется элемент, добавленный раньше всех. |
| **TTL-based** | Time To Live | Элемент удаляется по истечении заданного времени жизни. |
| **Random** | — | Вытесняется случайный элемент. |
| **allkeys-lru** | (Redis-политика) | LRU применяется ко всем ключам. |
| **volatile-lru** | (Redis-политика) | LRU применяется только к ключам с установленным TTL. |
| **allkeys-lfu** | (Redis-политика) | LFU применяется ко всем ключам. |
| **noeviction** | (Redis-политика) | При нехватке памяти возвращается ошибка вместо вытеснения. |

Redis/KeyDB позволяет задать политику через параметр `maxmemory-policy` в конфигурации.

### 4. Что такое и зачем используется Redis (KeyDB)?

**Redis** (Remote Dictionary Server) — это in-memory хранилище данных с открытым исходным кодом, работающее по принципу «ключ-значение». Поддерживает персистентность данных (RDB/AOF), репликацию, Pub/Sub, транзакции и Lua-скриптинг.

**KeyDB** — форк Redis, совместимый с его протоколом и API, но обеспечивающий многопоточную обработку команд (Redis традиционно однопоточный). KeyDB позволяет достигать более высокой пропускной способности на многоядерных серверах.

**Зачем используется:**
- Кэширование (уменьшение нагрузки на БД).
- Управление сессиями пользователей.
- Очереди задач (Lists как FIFO/LIFO).
- Pub/Sub и стриминг сообщений (Streams).
- Счётчики, рейт-лимитеры, лидерборды (Sorted Sets).
- Геопространственные индексы.
- Полнотекстовый поиск (через модуль RediSearch).

### 5. Какие есть основные типы данных в Redis (KeyDB)?

| Тип | Описание | Типичное применение |
|-----|----------|---------------------|
| **String** | Бинарная строка (или число). Максимум 512 МБ. | Кэш значений, счётчики, флаги. |
| **List** | Двусвязный список строк. | Очереди, стеки, история действий. |
| **Hash** | Словарь «поле → значение». | Хранение объектов (профиль пользователя). |
| **Set** | Неупорядоченное множество уникальных строк. | Теги, подписки, множественные операции. |
| **Sorted Set (ZSet)** | Множество с числовым score, отсортированное по нему. | Лидерборды, приоритетные очереди. |
| **Bitmap** | Битовый массив над строкой. | Флаги присутствия, HyperLogLog-подобные задачи. |
| **HyperLogLog** | Вероятностная структура для подсчёта уникальных значений. | Уникальные посетители. |
| **Stream** | Append-only лог с consumer groups. | Event sourcing, очереди сообщений. |
| **Geo** | Геопространственный индекс на ZSet. | Поиск ближайших объектов. |

### 6. Как можно взаимодействовать с Redis (KeyDB)?

1. **redis-cli / keydb-cli** — стандартный интерактивный клиент командной строки.
2. **Клиентские библиотеки** — для каждого языка есть официальные или популярные клиенты:
   - Go: `github.com/redis/go-redis/v9`
   - Python: `redis-py`
   - Java: `Jedis`, `Lettuce`
   - Node.js: `ioredis`
3. **RESP-протокол напрямую** — низкоуровневое TCP-соединение с текстовым протоколом RESP (Redis Serialization Protocol).
4. **HTTP REST API** — через сторонние прокси, например `webdis`.
5. **Язык Lua** — встроенное скриптирование через команду `EVAL` для атомарных транзакций.
6. **Redis Sentinel / Cluster API** — для управления отказоустойчивыми кластерами.

### 7. LIST — операции и асимптотика

**List** — двусвязный список; вставка/удаление с концов выполняется за O(1).

```bash
# Добавить элементы в начало списка (Left PUSH)
LPUSH student:{группа}:{номер}:timetable "Понедельник: Математика 09:00"
LPUSH student:{группа}:{номер}:timetable "Вторник: Алгоритмы 11:00"
LPUSH student:{группа}:{номер}:timetable "Среда: БД 13:00"

# Получить диапазон элементов (0 = первый, -1 = последний)
LRANGE student:{группа}:{номер}:timetable 0 -1
# → ["Среда: БД 13:00", "Вторник: Алгоритмы 11:00", "Понедельник: Математика 09:00"]

# Получить элемент по индексу
LINDEX student:{группа}:{номер}:timetable 0
# → "Среда: БД 13:00"

# Удалить count вхождений значения (0 = все)
LREM student:{группа}:{номер}:timetable 1 "Вторник: Алгоритмы 11:00"
```

| Команда | Асимптотика | Пояснение |
|---------|-------------|-----------|
| LPUSH / RPUSH | O(1) на элемент | Вставка в начало/конец списка |
| LRANGE | O(S+N) | S — смещение от начала; N — число возвращаемых элементов |
| LINDEX | O(N) | Обход до нужного индекса |
| LREM | O(N+M) | N — длина списка; M — число удалённых элементов |

### 8. STRING — операции и асимптотика

**String** — базовый тип Redis; хранит произвольные бинарные данные, числа, JSON.

```bash
# Установить строку
SET student:{группа}:{номер} "{ФИО}"

# Получить строку
GET student:{группа}:{номер}
# → "{ФИО}"

# Установить строку с TTL (секунды)
SET student:{группа}:{номер} "{ФИО}" EX 3600

# Атомарное увеличение числового значения
INCR some_counter
```

| Команда | Асимптотика | Пояснение |
|---------|-------------|-----------|
| SET | O(1) | Простая запись по ключу |
| GET | O(1) | Простое чтение по ключу |
| INCR / DECR | O(1) | Атомарная арифметика |
| APPEND | O(1) | Добавление к строке |
| STRLEN | O(1) | Длина хранится отдельно |

### 9. HASH — операции и асимптотика

**Hash** — словарь «поле → строка» внутри одного ключа. Удобен для хранения объектов.

```bash
# Установить несколько полей сразу
HSET student:{группа}:{номер}:info \
     name  "{ФИО}" \
     age   "21" \
     email "{email}@university.ru"

# Получить все поля и значения
HGETALL student:{группа}:{номер}:info
# → ["name", "{ФИО}", "age", "21", "email", "{email}@university.ru"]

# Получить одно поле
HGET student:{группа}:{номер}:info name
# → "{ФИО}"
```

| Команда | Асимптотика | Пояснение |
|---------|-------------|-----------|
| HSET | O(N) | N — число устанавливаемых полей |
| HGET | O(1) | Чтение одного поля |
| HGETALL | O(N) | N — число полей в хэше |
| HDEL | O(N) | N — число удаляемых полей |
| HEXISTS | O(1) | Проверка существования поля |

---

### 10. SET — операции и асимптотика

**Set** — неупорядоченное множество уникальных строк. Дубликаты игнорируются автоматически.

```bash
# Добавить элементы (дубль "Go" не добавится второй раз)
SADD student:{группа}:{номер}:skills "Go" "Python" "Docker" "Go"

# Получить все элементы
SMEMBERS student:{группа}:{номер}:skills
# → {"Go", "Python", "Docker"}  (порядок произвольный)

# Добавить ещё один дубль — проигнорируется
SADD student:{группа}:{номер}:skills "Go"
# → (integer) 0  (0 новых элементов добавлено)

# Удалить элемент
SREM student:{группа}:{номер}:skills "Python"
```

| Команда | Асимптотика | Пояснение |
|---------|-------------|-----------|
| SADD | O(N) | N — число добавляемых элементов |
| SMEMBERS | O(N) | N — размер множества |
| SREM | O(N) | N — число удаляемых элементов |
| SISMEMBER | O(1) | Проверка принадлежности |
| SUNION / SINTER | O(N*M) | Объединение/пересечение нескольких множеств |

---

### 11. ZSET — операции и асимптотика

**Sorted Set** — множество уникальных элементов, каждый из которых имеет числовой score. Элементы автоматически упорядочены по score.

```bash
# Добавить задачи с приоритетами
ZADD student:{группа}:{номер}:tasks_w_priority 10 "Сдать лабораторную работу 2"
ZADD student:{группа}:{номер}:tasks_w_priority 5  "Прочитать документацию Redis"
ZADD student:{группа}:{номер}:tasks_w_priority 20 "Подготовиться к экзамену"
ZADD student:{группа}:{номер}:tasks_w_priority 1  "Написать конспект"

# Получить все элементы по возрастанию score с выводом score
ZRANGE student:{группа}:{номер}:tasks_w_priority 0 -1 WITHSCORES
# → ["Написать конспект", "1", "Прочитать документацию Redis", "5", ...]

# Получить 1 элемент с наибольшим score (топ-1 по убыванию)
ZREVRANGE student:{группа}:{номер}:tasks_w_priority 0 0
# → ["Подготовиться к экзамену"]

# Увеличить score элемента на 15
ZINCRBY student:{группа}:{номер}:tasks_w_priority 15 "Написать конспект"
# → "16"  (новый score)

# Удалить элемент
ZREM student:{группа}:{номер}:tasks_w_priority "Прочитать документацию Redis"
```

| Команда | Асимптотика | Пояснение |
|---------|-------------|-----------|
| ZADD | O(log N) | N — размер множества; вставка в упорядоченное дерево |
| ZRANGE | O(log N + M) | M — число возвращаемых элементов |
| ZREVRANGE | O(log N + M) | То же, в обратном порядке |
| ZINCRBY | O(log N) | Обновление score и перемещение в дереве |
| ZREM | O(log N * M) | M — число удаляемых элементов |
| ZSCORE | O(1) | Получить score конкретного элемента |
| ZRANK | O(log N) | Получить позицию элемента |

### 12. Как реализуется работа с Redis в приложении?

В приложении используется клиент `github.com/redis/go-redis/v9`.

**Инициализация клиента** (`internal/cache/redis/cache.go`):

```go
client := redis.NewClient(&redis.Options{
    Addr:     "keydb:6379",
    Password: "",
    DB:       0,
})
// Проверка соединения
client.Ping(ctx).Err()
```

**Чтение из кэша:**

```go
raw, err := c.client.Get(ctx, key).Bytes()
// err == redis.Nil  →  ключ отсутствует (промах)
```

**Запись в кэш с TTL:**

```go
c.client.Set(ctx, key, data, 60*time.Second).Err()
```

**Инвалидация ключа:**

```go
c.client.Del(ctx, key).Err()
```

Кэш взаимодействует с приложением через интерфейс `Cache`, определённый в пакете `usecase`. Это позволяет подменять реализацию кэша без изменения бизнес-логики (принцип инверсии зависимостей).

### 13. Как реализуется стратегия cache-aside и когда она обоснована?

**Cache-aside (Lazy Loading)** — стратегия, при которой само приложение управляет взаимодействием между кэшем и хранилищем данных:

1. Приложение проверяет кэш по ключу.
2. Если данные есть (HIT) — возвращает их немедленно.
3. Если данных нет (MISS) — читает из основного хранилища (БД), сохраняет результат в кэш с TTL и возвращает данные клиенту.
4. При записи или удалении — инвалидирует соответствующий ключ кэша.

**Реализация в данном проекте** (`internal/usecase/usecase.go`):

```
GET /notes
  │
  ├─► Cache.Get("notes:all")
  │     ├─ HIT  → log "HIT"  → вернуть данные
  │     └─ MISS → log "MISS" → Repo.GetAll() → Cache.Set("notes:all", data, 60s) → вернуть данные
  │
POST /notes  →  Repo.Create()  →  Cache.Del("notes:all")
DELETE /notes/{id}  →  Repo.Delete()  →  Cache.Del("notes:all")
```

**Когда стратегия обоснована:**
- Данные читаются значительно чаще, чем обновляются (read-heavy workload).
- Допустима небольшая временная несогласованность между кэшем и БД (eventual consistency).
- Не все данные нужны в кэше — загружаются «по требованию».
- Простота реализации важнее минимальной задержки при первом запросе.

**Компромиссы:**
- «Холодный старт»: первый запрос всегда идёт в БД.
- При множестве одновременных промахов возможен «thundering herd» (шквал параллельных запросов к БД). Решение — distributed lock или single-flight.
- Данные в кэше могут устареть до истечения TTL, если их изменить напрямую в БД.

### Проверка созданных ключей

```bash
# Прочитать имя
GET student:{ваша_группа}:{номер}

# Прочитать все поля профиля
HGETALL student:{ваша_группа}:{номер}:info

# Получить одно поле
HGET student:{ваша_группа}:{номер}:info name

# Прочитать всё расписание

# Получить первый элемент расписания (индекс 0)

# Удалить один элемент расписания
LREM student:{ваша_группа}:{номер}:timetable 1 "Среда: Базы данных 13:00"

# Прочитать все навыки
SMEMBERS student:{ваша_группа}:{номер}:skills

# Добавить дублирующийся навык (вернёт 0 — не добавлен)
SADD student:{ваша_группа}:{номер}:skills "Go"

# Удалить навык
SREM student:{ваша_группа}:{номер}:skills "Python"

# Прочитать задачи по возрастанию приоритета с выводом score
ZRANGE student:{ваша_группа}:{номер}:tasks_w_priority 0 -1 WITHSCORES

# Получить задачу с наивысшим приоритетом
ZREVRANGE student:{ваша_группа}:{номер}:tasks_w_priority 0 0

# Добавить задачу
ZADD student:{ваша_группа}:{номер}:tasks_w_priority 8 "Провести ревью документации"

# Увеличить приоритет задачи
ZINCRBY student:{ваша_группа}:{номер}:tasks_w_priority 5 "Написать конспект лекций"

# Удалить задачу
ZREM student:{ваша_группа}:{номер}:tasks_w_priority "Прочитать документацию Redis"
```

## Bash-скрипт для сканирования ключей

Скрипт обходит все ключи с паттерном `student:*` без блокирующей команды `KEYS` (использует `SCAN` — безопасно для production):

```bash
#!/usr/bin/env bash
# scan_student_keys.sh
# Использование: ./scan_student_keys.sh [host] [port]
# По умолчанию: localhost 6379

HOST="${1:-localhost}"
PORT="${2:-6379}"
PATTERN="student:*"

echo "Scanning keys matching '${PATTERN}' on ${HOST}:${PORT} ..."
echo "-------------------------------------------------------------"

cursor=0
total=0

while true; do
    # SCAN возвращает: первая строка — новый cursor, остальные — ключи
    result=$(keydb-cli -h "$HOST" -p "$PORT" SCAN "$cursor" MATCH "$PATTERN" COUNT 100)
    cursor=$(echo "$result" | head -1)
    keys=$(echo "$result" | tail -n +2)

    if [ -n "$keys" ]; then
        while IFS= read -r key; do
            [ -z "$key" ] && continue
            type=$(keydb-cli -h "$HOST" -p "$PORT" TYPE "$key")
            ttl=$(keydb-cli  -h "$HOST" -p "$PORT" TTL  "$key")
            echo "KEY: $key  |  TYPE: $type  |  TTL: ${ttl}s"
            total=$((total + 1))
        done <<< "$keys"
    fi

    # cursor == 0 означает конец итерации
    [ "$cursor" -eq 0 ] && break
done

echo "-------------------------------------------------------------"
echo "Total keys found: $total"
```

Сделать скрипт исполняемым и запустить:

```bash
chmod +x scan_student_keys.sh
./scan_student_keys.sh localhost 6379
```

Пример вывода:

```
Scanning keys matching 'student:*' on localhost:6379 ...
-------------------------------------------------------------
KEY: student:{ваша_группа}:{номер}                    |  TYPE: string  |  TTL: -1s
KEY: student:{ваша_группа}:{номер}:info               |  TYPE: hash    |  TTL: -1s
KEY: student:{ваша_группа}:{номер}:timetable          |  TYPE: list    |  TTL: -1s
KEY: student:{ваша_группа}:{номер}:skills             |  TYPE: set     |  TTL: -1s
KEY: student:{ваша_группа}:{номер}:tasks_w_priority   |  TYPE: zset    |  TTL: -1s
-------------------------------------------------------------
Total keys found: 5
```

> TTL `-1` означает, что ключ не имеет срока истечения. Ключ `notes:all`, созданный приложением, будет иметь TTL `60s`.
